# 第二阶段试衣历史闭环设计

## 背景

第一阶段已经完成工程基线收敛：

- 文档和配置与真实技术栈一致
- 持久化模型、共享类型、预览态边界已经初步对齐
- API 路由和页面层的主要 lint 阻塞已清理
- 全量 `npm run lint` 已达到 `0 errors`

当前剩余的核心问题是业务链路没有闭环：

- `/tryon` 可以完成 AI 生成
- `/api/tryon-history` 已支持保存和读取历史
- `/profile` 已支持展示历史
- 但 `/tryon` 成功生成后没有手动保存入口，也没有和历史记录真正连起来

## 第二阶段目标

把 `登录用户 -> AI 试衣 -> 手动保存 -> 个人中心查看历史` 做成真实可用的闭环。

第二阶段完成后应满足：

- 未登录访问 `/tryon` 会被拦截并跳转登录页
- 登录用户可以正常完成 AI 试衣生成
- 生成成功后，结果页提供“保存到历史”操作
- 保存成功后，结果页立即切换为已保存状态，并提供进入个人中心的入口
- 用户进入 `/profile` 后能看到刚保存的试衣历史
- 如果保存失败，用户可以直接重试保存，不需要重新生成 AI 结果

## 非目标

第二阶段明确不做以下工作：

- 不支持匿名试穿
- 不做生成成功后的自动保存
- 不引入 OSS、COS、S3 等云存储体系
- 不重构图片持久化方案
- 不重做 `profile` 页面布局
- 不新增历史记录编辑、重命名或分类能力
- 不解决 `no-img-element` 警告

## 交互决策

### 登录边界

`/tryon` 在进入流程前先校验当前用户是否登录。

- 已登录：正常进入试衣流程
- 未登录：直接跳转 `/auth/login`

本阶段不提供匿名试穿，也不允许未登录用户先生成、后补登录保存。

### 保存策略

AI 生成成功后不自动保存。

用户需要在结果页主动点击“保存到历史”：

- 这样可以把“生成成功”和“历史写库成功”拆开处理
- 如果保存失败，用户仍然保留当前结果图，可以重试保存
- 避免因为历史写入失败而破坏整个试衣流程体验

### 保存后状态

保存成功后，结果页需要明确反馈：

- “保存到历史”按钮变为不可重复触发的已保存状态
- 展示“去个人中心查看”入口
- 本轮结果仍保留在页面，不强制跳转

## 数据流

### 1. 进入 `/tryon`

页面初始化时请求 `/api/auth/me`：

- 返回用户信息：继续加载试衣流程
- 返回未登录：前端跳转到 `/auth/login`

### 2. 生成 AI 结果

沿用现有流程：

- `POST /api/ai-tryon` 创建异步任务
- `GET /api/ai-tryon/status` 轮询状态
- 成功时把 `resultImageUrl` 写入页面状态

### 3. 手动保存历史

用户点击“保存到历史”后，请求 `POST /api/tryon-history`，提交：

- `personImageUrl`
- `clothImageUrl`
- `keepClothImageUrl`
- `resultImageUrl`
- `clothType`
- `tryOnMode`

本阶段 `keepClothImageUrl` 继续允许为 `null`。

### 4. 保存成功后的前端同步

保存成功后，前端状态更新为：

- `isSavingHistory = false`
- `historySaved = true`
- `historySaveError = null`

保存失败后：

- 保留已生成的 `resultImageUrl`
- 记录错误信息并允许重试保存
- 不重置步骤，不重跑 AI 任务

## 页面与接口设计

### `/tryon` 页面

需要新增三类状态：

- 登录态加载状态
- 历史保存状态
- 历史保存错误状态

页面职责变成三段：

1. 登录守卫
2. AI 结果生成
3. 历史保存与成功反馈

### `ResultView` 组件

需要从“只负责展示结果/重试”扩展为“结果展示 + 保存动作反馈”。

建议新增能力：

- 显示“保存到历史”按钮
- 保存中显示 loading 状态
- 已保存时显示禁用或成功态按钮
- 提供“去个人中心查看”入口
- 保存失败时显示错误提示并允许重试保存

### `POST /api/tryon-history`

现有路由基本满足需要，第二阶段只做必要的小修正：

- 与当前 `verifyToken` 返回类型对齐
- 保持参数校验清晰
- 不改变数据表结构

### `/profile`

本阶段不重写历史列表。

只要历史接口仍按创建时间倒序返回，用户保存后再进入 `/profile` 就能立即看到新纪录。若本地前端状态需要小修补以确保首次加载稳定，可做小范围调整，但不扩大为页面重构。

## 受影响文件范围

优先修改以下文件：

- `app/src/app/tryon/page.tsx`
- `app/src/components/tryon/ResultView.tsx`
- `app/src/app/api/tryon-history/route.ts`
- `app/src/app/profile/page.tsx`

如为保证登录守卫和保存反馈一致性，需要极小范围补充本地类型或共用辅助逻辑，可以加在现有文件内，避免新建复杂目录结构。

## 验收标准

第二阶段验收以以下事实为准：

1. 未登录访问 `/tryon` 会跳转到 `/auth/login`
2. 登录用户可以完成 AI 生成
3. 生成成功后页面出现“保存到历史”
4. 保存成功后结果页显示已保存状态和进入个人中心入口
5. 保存失败时可以直接重试保存，不需要重新生成
6. `/profile` 能看到新保存的试衣记录
7. 相关代码变更后 `npm run lint` 仍保持 `0 errors`

## 风险与控制

### 风险 1：保存动作与 AI 生成状态耦合过深

控制方式：把“生成结果状态”和“历史保存状态”拆成独立状态字段。

### 风险 2：未登录用户在页面初始化时出现闪烁

控制方式：在 `/tryon` 页面加入显式登录检查加载态，再决定渲染或跳转。

### 风险 3：保存失败迫使用户重新试穿

控制方式：保存失败时不清空 `resultImageUrl`，只允许重试保存。
