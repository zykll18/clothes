# 奢侈品叙事风格 AI 试衣首页设计

## 背景

当前首页仍然是典型原型站形态：

- 蓝白渐变背景与通用功能卡视觉
- 文案直接描述功能，但缺少品牌记忆点
- 导航、首屏、功能区块之间没有统一的材质语言
- 与项目希望传达的 `高级时装 + AI试衣` 方向明显不一致

用户已经提供一张黑底银白发光花朵图作为核心视觉素材，并明确希望它成为整站统一视觉母题，而不是只在首屏作为一次性背景图使用。

同时，用户参考了一类“电影感 landing page”的结构和材质语言，希望借用其：

- 黑底高对比氛围
- liquid-glass 玻璃质感
- 慢节奏、克制的入场动画
- 强首屏叙事和编辑感排版

但项目最终仍然要卖的是 `AI 试衣产品`，不能做成纯概念艺术页。

## 目标

将首页重构为“奢侈品广告页外壳中的 AI 试衣产品页”，实现以下目标：

1. 用户进入首页后，首先感受到高级时装品牌调性
2. 用户在 3 到 5 秒内明确理解这是 `AI 试衣 / 穿搭预览` 产品
3. 提供清晰但不过度功能化的主路径，引导用户进入注册或试用
4. 让花朵素材升级为整站视觉母题，贯穿首页而不被机械重复

## 非目标

本次首页设计明确不包含：

- 不重做认证、预览、个人中心等内页
- 不重构 API、数据库或业务流程
- 不把首页扩展成完整品牌官网
- 不引入与当前工程不匹配的 CDN 脚本式架构
- 不逐字照搬参考 prompt 中的太空题材、视频资源和组件实现

## 推荐方案

采用“产品叙事平衡”路线：

- 首页先以品牌视觉和叙事切入
- 首屏即明确点出 `AI 试衣`
- 后续区块再用时装 campaign 的语言解释产品能力和使用路径

相比“品牌大片优先”，该方案不会牺牲产品清晰度；相比“功能展示优先”，它能最大化利用花朵素材建立品牌识别。

## 核心体验原则

### 1. 母题不是贴图，而是材质来源

花图只在首屏完整出现一次。后续区块不再重复整张图，而是提取它的语言：

- 花瓣弧线
- 冷白高光
- 银色反射
- 柔和 bloom 光晕
- 黑底中的雕塑感留白

这样用户会感到整站统一，而不是在不同位置看到同一张图被反复复制。

### 2. 奢侈品气质优先，但不能牺牲产品理解

文案、排版和留白按高级时装 campaign 的方式处理，但每一屏都必须服务于以下问题之一：

- 这是什么产品
- 为什么值得试
- 怎么使用
- 下一步该做什么

### 3. 动效必须慢、轻、克制

不做炫技式大幅运动，不做科技产品常见的强渐变和激烈发光。所有动态效果服务于“呼吸感”和“镜头感”：

- 文字轻微 blur-in
- 按钮和玻璃层轻微反馈
- 背景光感缓慢变化
- 区块进入保持低速度和高留白

## 视觉系统

### 色彩

整站固定为以下体系：

- 主背景：`#050505` 到 `#0b0b0b`
- 主文字：柔白，不用纯蓝白高科技色
- 次级文字：半透明暖冷中性的白灰
- 高光：银白、冷白
- 分隔和边缘：低透明度白色描边

禁止回退到当前首页的蓝白渐变体系，也不使用紫色、霓虹蓝等“AI 默认色”。

### 字体

- 标题字体：高时装感 serif，斜体使用为主，用于情绪和叙事
- 正文字体：干净的 sans-serif，用于说明、导航和按钮
- 数字和辅助说明：延续 sans-serif，不做过多风格混搭

标题应具备品牌大片气质，但正文必须保持清晰可读。

### 材质

所有可交互表层统一为同一类 liquid-glass 语言：

- 导航胶囊
- 主按钮
- 次要按钮
- 信息卡
- 标签 chip

层级区分通过透明度、blur 强度、描边亮度和内阴影深浅完成，不依靠五花八门的颜色区分。

### 构图

整体布局遵循编辑式排版，而不是居中堆叠：

- 首屏：左侧文字，右侧花图
- 第二、三屏：大留白、偏左对齐或不对称构图
- 卡片区：有秩序，但避免传统 SaaS 卡片墙

## 首页信息结构

首页采用 5 个主区块。

### 1. Hero

目标：先建立时装气质，再明确产品身份。

布局：

- 顶部悬浮玻璃导航
- 左侧：品牌叙事标题、副文案、主次 CTA
- 右侧：花图作为首屏核心雕塑视觉
- 首屏底部：轻量品牌信任信息或简化标签

内容要求：

- 标题带叙事感，但明确指向 `AI 试衣`
- 副文案解释价值，不堆技术术语
- 主按钮指向开始试用或注册
- 次按钮可指向效果预览或体验介绍

### 2. Brand Statement

目标：把用户从“这很美”推进到“这和穿搭选择有关”。

形式：

- 一段短而有节奏的品牌说明
- 大留白、细字、局部弧线或柔光元素
- 不用功能卡，不做教程式解释

内容语义：

强调穿搭决策、试穿感受、风格确认和数字化体验，而不是泛泛地说“AI 改变未来”。

### 3. How It Works

目标：用高级方式讲清产品路径。

结构：

- `上传人物`
- `选择服装`
- `生成试穿`

表达方式：

- 不是蓝色数字圆点流程
- 更像时装画册中的三段说明页
- 每段搭配简短标题、说明、极简视觉符号或局部图形

### 4. Capability Cards

目标：解释产品能力，但继续维持品牌材质统一。

候选能力：

- 真实轮廓
- 快速换装
- 风格比较

卡片策略：

- 使用 glass 材质，但避免“企业功能九宫格”
- 卡片内部信息层级清楚，文字短
- 每张卡片都像陈列品，而不是开发文档条目

### 5. Closing CTA

目标：完成收束和转化。

形式：

- 一句有记忆点的收尾文案
- 一个明确的主 CTA
- 不重复长段说明

视觉上应更简洁，让用户在前文建立信任后自然行动。

## 花图使用策略

### 首屏中的使用

- 作为首屏右侧核心视觉
- 保留花头、花瓣高光和花茎弧线的雕塑感
- 不机械全屏铺满
- 通过遮罩、渐隐和局部压暗处理左上角干扰文字区域

### 后续区块中的转译

后续区块不再使用整张花图，而使用以下衍生语言：

- 花瓣弧线式分隔
- 冷白 bloom 光晕
- 银色高光边缘
- 柔和玻璃反射
- 纵向弧线延展页面节奏

### 响应式规则

- 桌面端：保留花的完整主轮廓，靠右布局
- 平板端：保留花头与部分花茎，压缩空白
- 移动端：优先保留花头，不强行展示整根花茎，避免裁切后画面失衡

## 组件与实现边界

首页实现优先在以下范围内收敛：

- `src/app/page.tsx`
- `src/components/layout/Navbar.tsx`
- 新增首页专用展示组件目录，例如 `src/components/home/*`
- `src/app/globals.css`
- 如需正式使用图片素材，可将素材复制到 `public/` 下的新路径中

实现上遵循当前项目技术栈：

- Next.js App Router
- React 19
- Tailwind 4
- 已安装 `lucide-react`

首版实现不依赖参考 prompt 的 CDN React、Babel 或视频交叉淡入架构。若后续需要更复杂动画，再单独评估是否引入新依赖。首版应优先使用当前仓库已有能力完成：

- CSS 渐隐/blur 入场
- 轻量悬停反馈
- 玻璃质感样式系统
- 响应式构图

## 导航设计

导航从当前白底产品栏切换为悬浮式处理：

- 背景透明或半透明
- 中央或偏左为胶囊式导航组
- 品牌标识简化、克制
- 登录/注册按钮与整站材质统一

导航的作用是维持高级感和方向感，不承担复杂信息架构。

## 文案方向

文案采用“有叙事感的高级时装 campaign”风格，但首页必须明确卖点，不走纯概念路线。

文案要求：

- 句子短
- 节奏感强
- 不堆营销形容词
- 不用技术黑话
- 在首屏副文案中明确点出 AI 试衣价值

## 可访问性与可用性

虽然整体走品牌化视觉，但需满足基本可用性：

- 主要文字与背景保持足够对比
- CTA 明确可识别，不做伪按钮
- 移动端首屏按钮区域不能被视觉图挤压
- 导航和内容顺序应支持键盘访问

## 验收标准

首页重构完成后，应满足以下事实：

1. 首页视觉从原型风格切换为统一的奢侈品叙事风格
2. 花图被成功转化为整站视觉母题，而不是仅一张背景图
3. 用户进入首页后能在短时间内明确理解这是 `AI 试衣` 产品
4. 首页结构完成 `情绪 -> 理解 -> 能力 -> 行动` 的叙事节奏
5. 导航、按钮、信息卡和文案在材质和语气上保持一致
6. 桌面端与移动端都具备稳定构图，不出现主视觉裁坏或信息层级混乱

## 测试与验证策略

实现阶段至少验证以下内容：

1. 本地视觉检查：桌面端与移动端首屏构图是否稳定
2. 交互检查：导航、CTA、悬停反馈是否一致
3. 回归检查：确保未破坏现有路由入口和页面基础可用性
4. 质量检查：`npm run lint`

若实现过程中加入额外动画或新依赖，需要额外验证：

- 首屏性能是否明显下降
- 文字可读性是否被背景和发光干扰

## 风险与控制

### 风险 1：过度追求品牌感，导致产品身份不清

控制方式：首屏副文案和主 CTA 必须明确指出这是 `AI 试衣` 产品。

### 风险 2：花图被重复使用，形成廉价贴图感

控制方式：整图只完整出现一次，后续只抽取语言，不重复摆图。

### 风险 3：照搬参考 prompt，导致实现与项目工程脱节

控制方式：只借结构、材质和节奏，不引入 CDN 模式和无关题材实现。

### 风险 4：落地后又回到普通 SaaS 组件堆叠

控制方式：所有分区先服从叙事，再补产品信息；避免功能卡泛滥和蓝色产品风回潮。

- 首页改造的分步实施顺序

- 视觉样式抽离策略
- 验证步骤与风险回归点
