# 第一阶段基础对齐设计

## 背景

当前项目已经具备穿搭预览、认证、衣橱管理、AI 试衣、试衣历史等原型能力，但工程层面存在明显漂移：

- 文档、环境变量示例与真实技术栈不一致
- Prisma schema、前端类型、API 返回结构不是同一套模型
- 若干脚手架残留和无效入口仍暴露在用户界面中
- `npm run lint` 不能通过，阻塞后续稳定开发

这导致后续继续开发业务闭环时，会不断在类型、字段和页面假设之间来回修补。

## 第一阶段目标

把仓库整理到一个稳定开发基线，使第二阶段可以直接实现 `衣橱 -> AI试衣 -> 历史记录` 的业务闭环，而不是继续处理基础设施漂移。

第一阶段完成后应满足：

- `npm run lint` 通过
- README 和 `.env.example` 反映真实项目状态
- 持久化模型、TypeScript 类型、API 返回结构不再明显冲突
- 默认脚手架残留和明显无效入口被移除或接通
- 开发基线明确为 `SQLite + Prisma`，并在文档中说明未来可迁移到 PostgreSQL，但当前实现不假定已经迁移

## 非目标

第一阶段明确不包含以下工作：

- 不重做整体 UI 风格
- 不大规模拆分页面或做体系化组件重构
- 不补完整业务闭环
- 不引入新的鉴权方案、上传方案或状态管理方案
- 不为未来 PostgreSQL 迁移提前引入复杂适配层

## 推荐方案

采用“模型先对齐”的收敛策略：

1. 以 `prisma/schema.prisma` 作为持久化模型的唯一事实源
2. 以当前真实 API 行为为页面消费边界
3. 把仅用于前端交互的临时字段从通用实体类型中分离
4. 在不扩大 UI 范围的前提下，修复会阻塞开发的工程质量问题

该方案相比“最小修补”更稳，因为它能直接降低第二阶段返工；相比“顺手重构”更收敛，不会把第一阶段拉成一次大修。

## 架构边界

第一阶段之后，项目的边界将分成两层：

### 1. 持久化数据层

由 Prisma schema 和 API 路由定义。任何需要落库的数据字段，都必须能在 schema 和 API 中找到明确来源。

### 2. 前端交互层

仅用于页面渲染、拖拽预览、临时编辑状态等短生命周期场景。此类字段不能再伪装成数据库实体字段，而应使用单独的前端类型承载。

这样做的结果是：

- 改数据库字段时，有清晰的修改路径
- 页面不会再默认依赖数据库中不存在的属性
- 后续补业务闭环时，前后端边界更清楚

## 数据与类型对齐策略

### Prisma 作为后端基线

`prisma/schema.prisma` 是后端持久化结构的唯一事实源。第一阶段不扩大 schema 范围，只在必要时做最小修正，保证它与当前 API 和页面使用方式一致。

### 应用类型收敛

`src/types/index.ts` 中的核心实体类型需要收敛到真实存在的数据：

- `ClothingItem` 不再混入 `position`、`scale`、`rotation`、`opacity` 这类纯前端预览字段
- `tags` 的类型应与真实存储方式一致，避免前端假定其为数组而后端实际为字符串
- `UserPreference` 若当前并未形成稳定使用路径，则降级为保守类型或移除未使用的误导性定义
- 通用 `ApiResponse` 去掉不必要的 `any`

### 预览态单独建模

`preview` 页面的拖拽、缩放、旋转、透明度等信息属于前端交互态，不属于通用衣物实体。第一阶段会为这类场景保留独立类型，例如：

- 基础衣物数据类型
- 带有预览变换信息的前端展示类型

这样可以保留页面能力，同时不污染通用业务模型。

## API 对齐策略

第一阶段不重写 API 体系，只做以下收敛：

- 为 `clothing`、`ai-tryon` 等路由去掉直接导致 lint 失败的 `any`
- 把请求和响应结构写成显式类型，减少隐式字段假设
- 保持现有业务语义不变，不借第一阶段偷偷引入新流程
- 页面层只消费 API 实际返回的数据，不再假设“未来会有”的字段

## 工程质量策略

第一阶段会修复所有当前直接阻塞 `eslint` 通过的问题，重点包括：

- `Input` 组件中 render 阶段的 `Math.random()` 纯度问题
- 各页面与 API 中未使用变量和显式 `any`
- 当前确实阻塞 lint 的类型定义问题

对 `img` 与 `next/image` 的问题，本阶段遵循“够用即可”的原则：

- 若只是警告且不阻塞当前目标，可在不扩大范围的前提下局部保留
- 若某处顺手修复成本很低，可一并处理

## 文档与配置策略

README、环境变量示例和项目元数据会更新为真实状态：

- 技术栈更新为 `Next 16 / React 19 / Tailwind 4 / Prisma / SQLite / DashScope`
- 开发说明围绕本地 SQLite 工作流编写
- `.env.example` 补充 AI 试衣实际依赖的配置，如 `DASHSCOPE_API_KEY`
- `layout` 元数据替换默认脚手架内容，反映真实项目名称和描述

第一阶段不会编写完整部署文档，只会保证本地开发说明真实可用。

## 导航与残留处理策略

第一阶段会处理“明显错误但不需要重构”的入口问题：

- 移除或改正当前不存在的 `/wardrobe` 导航入口
- 保留现有页面结构，不引入新的路由体系
- 清除默认脚手架元数据和其他误导性残留

原则是：不做新的导航设计，只消除明显无效信息。

## 验收标准

第一阶段验收以以下事实为准：

1. `npm run lint` 通过
2. 文档能正确说明如何在本地启动项目
3. 环境变量示例与 AI 试衣、JWT、本地数据库的真实依赖一致
4. 类型定义与 Prisma schema/API 不再存在明显结构漂移
5. 默认脚手架元数据和无效导航入口已处理

## 风险与控制

### 风险 1：为追求“类型完美”而扩展 schema 范围

控制方式：第一阶段只做对齐，不做功能性扩表；前端临时态用单独类型承载。

### 风险 2：顺手把第一阶段做成 UI 重构

控制方式：只处理无效入口、默认元数据和直接阻塞 lint 的代码，不重做视觉设计。

### 风险 3：文档仍然滞后于实现

控制方式：README 和 `.env.example` 更新以当前代码真实行为为准，不写“未来状态”。

## 实施后的下一步

第一阶段完成后，再进入第二阶段，专注打通：

- 衣橱数据被 AI 试衣真实消费
- AI 结果真实写入试衣历史
- 页面间主流程从“原型可演示”变成“真实可闭环”

第二阶段以本设计产出的稳定开发基线为前提，不在其中重复处理第一阶段问题。
